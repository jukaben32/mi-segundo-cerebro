# Autoresearch Optimizer Skill

This is an experimental skill designed to let you, the AI, perform autonomous research and optimization on a target task (like code improvement, prompt engineering, marketing copy, etc.).

## 🚀 The Core Concept
The system follows a strict biological-like evolution loop: 
1. **Current Baseline**: The best known working version of the asset.
2. **Challenger Mutation**: A new variant generated by you (the LLM) to attempt to beat the baseline.
3. **Evaluation**: Testing the challenger against a defined metric or test script.
4. **Promotion**: If the challenger wins, it becomes the new baseline.
5. **Logging & Notification**: Record the results to a log file and (optionally) ping a webhook.
6. **Repeat**: Loop indefinitely.

---

## 🛠️ Step 1: Setup Phase

When you are invoked with this skill, work with the user to establish the following:

1. **Target Asset**: What are we optimizing? (e.g., `prompt.txt`, `model.py`, `email_copy.md`).
2. **Evaluation Metric**: How will we know if a variant is better? (e.g., "Run `test.py` and grep the `accuracy` score", "Use an LLM-as-a-judge to evaluate the flow").
3. **Log File**: Ensure a `results.tsv` exists in the working directory. If it doesn't, create it with headers: `timestamp	commit	variant_name	score	status	description`.
4. **Webhook URL**: (Optional) Ask the user if they want a webhook ping notification via `curl` when a challenger beats a baseline.

If these aren't provided by the user upfront, **ask for them** before beginning the loop.

---

## 🔬 Step 2: The Continuous Experimentation Loop

Once setup is confirmed, begin the autonomous loop. **DO NOT PAUSE** to ask the human if you should continue after each run. Run indefinitely until interrupted or encountering an unrecoverable crash.

### Loop Cycle:

1. **Analyze Baseline**: Look at the current best asset and its past results in `results.tsv`.
2. **Hypothesize**: Formulate a hypothesis on how to improve the metric. (e.g., "Changing the activation function to SwiGLU might lower validation loss", "Adding urgency to the email subject might increase open rates").
3. **Mutate (Create Challenger)**: 
   - Modify the target asset file.
   - Keep your changes small and atomic so you can easily attribute success or failure to a specific mutation.
4. **Evaluate**: 
   - Run the predefined evaluation script or method.
   - Wait for the results. (e.g., extracting `val_bpb` or `accuracy`).
5. **Determine Outcome**:
   - **Win**: The challenger metric is better than the baseline. 
     - **Action**: Commit the changes. This is the new baseline. Log `keep` in `results.tsv`. 
     - **Ping**: If a webhook is configured, run `curl -X POST -H "Content-Type: application/json" -d '{"text": "🚀 New Baseline Achieved! Variante: [name] Score: [score]"}' [WEBHOOK_URL]`.
   - **Loss/Tie**: The challenger metric is worse or identical to the baseline.
     - **Action**: Discard changes (revert the file back to the baseline state). Log `discard` in `results.tsv`.
   - **Crash/Error**: The challenger broke the script or failed.
     - **Action**: Fix trivial typos if obvious. Otherwise, revert the file to baseline. Log `crash` in `results.tsv`.
6. **Repeat**: Move to Step 1.

---

## 📜 Logging Format (`results.tsv`)

Keep a meticulous record of every run. Append each experiment to `results.tsv`.

Format: (Tab separated, do *not* use commas)
```tsv
timestamp	commit	variant_name	score	status	description
2026-03-30T10:00:00	abc1234	baseline_1	0.9950	keep	Initial starting point
2026-03-30T10:05:00	-	challenger_1_lr	0.9980	discard	Increased learning rate
2026-03-30T10:10:00	def5678	challenger_2_swish	0.9810	keep	Changed to Swish activation
```

## ⚠️ Important Rules for the Agent

- **Never Stop**: Once the loop starts, run it autonomously in the background. Do not say "Let me know when you want to continue." Just continue.
- **Simplicity Wins**: If a challenger achieves an identical score but with simpler code/copy, promote it to base.
- **Isolate Variables**: Change only one or two sensible parameters per challenger to learn efficiently.
- **Time Boxing**: If an evaluation run takes too long (e.g. > 10 minutes), kill it and mark it as a `crash`.
